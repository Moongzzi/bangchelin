당신은 React + TypeScript 기반 프로젝트의 시니어 프론트엔드 엔지니어이자,
기존에 생성된 페이지와 공통 컴포넌트를 점검/정리하면서 실제 서비스 화면을 완성하는 UI 구현 어시스턴트입니다.

현재 프로젝트는 Bangchelin Guide입니다.
카카오톡 오픈채팅 “방탈출 미식 협회” 전용 커뮤니티 웹페이지이며,
현재는 4단계 페이지 제작 중입니다.

이번 작업의 목표는 /calendar 경로에 들어갈 일정 관리 페이지, 즉 CalendarPage.tsx를 구현하는 것입니다.

첨부한 첫 번째 이미지는 사용자가 /calendar 페이지에 처음 접속했을 때 보이는 기본 상태 화면입니다.
첨부한 두 번째 이미지는 일정 생성, 날짜 선택, 상세 패널 표시, 모달 표시, 확인/취소 팝업 등 기능 동작에 따른 전체 화면 흐름을 표현한 참고 이미지입니다.

중요:
이번 작업은 단순히 이미지를 보고 비슷하게 만드는 것이 아니라,
기존 프로젝트 정의, 디자인 토큰, 공통 컴포넌트, 이전 페이지 제작 방식과 이어지도록
실제 서비스 구조에 맞는 CalendarPage를 구현하는 것입니다.

기존 Agent가 이미 CalendarPage.tsx 또는 유사 파일을 임의 생성했을 가능성이 있으므로,
무조건 새로 만들지 말고 기존 구조를 점검하고,
재사용 가능한 부분은 유지하되 시안과 맞지 않는 구조는 정리/리팩터링하세요.

이미지에서 명확히 확인 가능한 사실:
- 상단에는 공통 Header가 있음
- 하단에는 공통 Footer가 있음
- 전체 배경은 밝은 베이지 톤임
- 좌측에는 오늘 날짜와 오늘 일정 개수를 보여주는 요약 패널이 있음
- 좌측 패널 상단에는 날짜 숫자, 요일, “오늘 일정” 개수가 표시됨
- 좌측 패널에는 “+ 일정 생성” 버튼이 있음
- 좌측 패널에는 오늘의 일정 카드 리스트가 있음
- 일정 카드에는 상태 배지, 카테고리 배지, 인원 수, 제목, 시간, 장소 정보가 있음
- 중앙에는 월간 캘린더가 크게 배치됨
- 캘린더 상단에는 연도, 월 이름, 이전/다음 월 이동 버튼이 있음
- 요일 헤더는 SUN ~ SAT로 표시됨
- 날짜 셀 안에 일정이 pink pill 또는 dot 형태로 표시됨
- 선택된 날짜는 배경 또는 border로 강조됨
- 일부 날짜 숫자는 일요일/토요일 컬러가 다르게 보임
- 기능 흐름 이미지에는 일정 생성 모달, 날짜 선택 UI, 확인 팝업, 우측 상세 패널이 나타남
- 우측 상세 패널은 특정 날짜 또는 특정 일정을 선택했을 때 화면 오른쪽에 표시되는 구조로 보임

텍스트 설명으로 보강된 사실:
- 첫 번째 이미지는 페이지 접속 시 기본 상태 화면임
- 두 번째 이미지는 기능 동작에 따른 화면 변화 흐름 전체를 표현한 참고 이미지임
- 현재는 mock data로 UI와 사용자 흐름을 먼저 만드는 단계임
- 실제 API 연결은 추후 단계에서 진행함

구현을 위해 합리적으로 추정하는 사실:
- CalendarPage는 Header + main calendar layout + Footer 구조로 조립하는 것이 적절함
- 좌측 패널, 월간 캘린더, 우측 상세 패널, 일정 생성 모달은 페이지 전용 컴포넌트로 분리하는 것이 적절함
- 일정 데이터는 mock 파일로 분리하고, 추후 Supabase API 응답 구조로 교체 가능하게 타입을 명확히 두는 것이 좋음
- 날짜 계산은 외부 라이브러리 없이 기본 Date 유틸 함수로 우선 구현 가능함
- 일정 상태값은 모집, 마감 등으로 관리 가능함
- 일정 카테고리는 방탈출, 연극, 보드게임, 기타 등으로 확장 가능함
- 일정 생성 모달은 공통 Modal이 있으면 재사용하고, 없다면 페이지 전용 최소 Modal로 작성함
- 삭제/저장 확인 팝업은 공통 Popup이 있으면 재사용하고, 없다면 최소 ConfirmDialog를 작성함

[Calendar 구현 방식 관련 추가 규칙]
--------------------------------------------------

이번 CalendarPage 구현에서는
반드시 직접 달력 로직을 처음부터 전부 구현할 필요는 없습니다.

프로젝트 구조, 유지보수성, 접근성, 일정 렌더링 복잡도 등을 고려했을 때
더 적절하다고 판단되면 외부 Calendar 라이브러리 또는 date 관련 라이브러리 사용을 허용합니다.

다만 아래 원칙을 반드시 지켜주세요.

1. 라이브러리 선택 이유를 먼저 설명할 것
2. 현재 프로젝트 구조와 디자인 시스템에 자연스럽게 통합 가능한 라이브러리를 우선 고려할 것
3. React + TypeScript strict mode와 호환되어야 함
4. 과도하게 무거운 라이브러리는 피할 것
5. 커스터마이징 가능한 구조를 우선할 것
6. 현재 시안(UI)을 구현 가능해야 함
7. 기존 공통 컴포넌트 및 디자인 토큰과 충돌하지 않아야 함
8. 추후 Supabase API 연결과 상태 관리 확장이 쉬워야 함

사용 가능 예시:
- FullCalendar
- react-big-calendar
- date-fns
- dayjs

단,
- drag & drop
- 실시간 협업
- Google Calendar 연동
- 과도한 scheduler 기능
등 현재 요구사항에 없는 기능은 추가하지 마세요.

또한 라이브러리를 사용할 경우:
- 왜 선택했는지
- 현재 요구사항에서 어떤 부분을 단순화하는지
- 어떤 부분은 커스텀 구현이 필요한지
를 먼저 설명한 뒤 구현을 진행하세요.

가능하면 “날짜 계산 로직은 라이브러리 활용 + UI는 커스텀 구현” 방향을 우선 고려해주세요.

라이브러리를 사용하지 않는 경우에는
기존 요구사항대로 Date 기반 직접 구현을 진행해도 됩니다.

기존 CalendarPage 또는 calendar 관련 파일이 있다면 다음 기준으로 점검하세요.

유지 가능한 부분:
- Header/Footer를 포함한 전체 페이지 조립 구조
- 기존 공통 Button/Input/Dropdown/Modal 컴포넌트 사용부
- mock data 구조 중 CalendarEvent와 유사한 타입
- 라우터에 /calendar 경로가 이미 등록되어 있다면 해당 구조

수정 필요한 부분:
- 시안과 다른 레이아웃
- 달력 grid가 실제 월간 구조가 아닌 단순 카드 나열인 경우
- 날짜 선택/일정 선택 상태가 분리되지 않은 경우
- mock data가 컴포넌트 내부에 하드코딩된 경우
- 색상값이 토큰 없이 직접 하드코딩된 경우
- 모달/상세 패널이 상태 기반으로 동작하지 않는 경우

재구성 필요한 부분:
- 달력 페이지 전체가 하나의 거대한 컴포넌트인 경우
- 날짜 계산 로직과 UI 렌더링이 과하게 섞인 경우
- any 타입 사용
- 실제 API 호출과 mock data가 섞여 있는 경우
- 시안에 없는 과한 기능이 포함된 경우

- Modal/ConfirmDialog가 없다면 이번 페이지 전용 최소 컴포넌트로 작성

기존 컬러 토큰을 우선 사용하세요.

시안 기준 스타일 방향:
[배경]
- 페이지 배경 : bg/default
- 달력배경 : bg/default
- 좌/우측 패널 배경 : surface/default

[텍스트]
- 기본 텍스트 컬러 : text/primary
- 일정 생성 버튼 텍스트 : text/on-primary

임의 색상을 새로 만들지 말고, 가능한 토큰을 사용하세요.
어떤 색상인지 애매한 경우, 임의로 만들거나 하드코딩하지 말고 질의하거나, 가능한 토큰 중 제일 유사한 토큰을 사용해주세요.

--------------------------------------------------
[접근성 요구사항]
--------------------------------------------------

- main 태그 사용
- 달력은 section 또는 table/grid 구조 중 하나로 접근성 고려
- 날짜 버튼은 button으로 구현
- 일정 항목도 클릭 가능하면 button 또는 a 사용
- 이전/다음 월 버튼에는 aria-label 제공
- 모달에는 role="dialog", aria-modal="true" 적용
- 확인 팝업에도 role="alertdialog" 적용
- 입력 필드는 label 연결
- ESC로 모달 닫기 가능하면 좋음
- 버튼 focus-visible 스타일 제공
- 색상만으로 상태를 구분하지 말고 텍스트 라벨도 포함

--------------------------------------------------
[액션 연결 요구사항]
--------------------------------------------------

반드시 아래 액션 연결 포인트가 코드상 드러나야 합니다.

1. 이전 월 버튼 클릭
   - currentMonth를 이전 달로 변경

2. 다음 월 버튼 클릭
   - currentMonth를 다음 달로 변경

3. 날짜 클릭
   - selectedDate 변경
   - 해당 날짜 강조
   - 우측 패널에 해당 날짜 일정 표시 가능

4. 일정 클릭
   - selectedEventId 변경
   - EventDetailPanel 표시

5. “+ 일정 생성” 클릭
   - isCreateModalOpen = true

6. 일정 생성 모달 저장 클릭
   - mock create handler 실행
   - TODO: Supabase API 연결 주석
   - 저장 후 모달 닫기 또는 ConfirmDialog 표시

7. 일정 생성 모달 취소 클릭
   - 닫기 또는 ConfirmDialog 표시

8. 일정 삭제 클릭
   - ConfirmDialog 표시
   - 확인 시 mock 삭제
   - TODO: Supabase API 연결 주석

9. Header nav의 캘린더 메뉴
   - /calendar 경로와 연결되어 있어야 함

[코드 작성 규칙]
--------------------------------------------------

- React + TypeScript strict mode 기준
- any 사용 금지

- props와 state 이름은 의미 있게 작성
- 날짜 계산 로직은 utils로 분리
- label, status, category 매핑은 constants로 분리
- 반복 문자열은 상수화
- CSS Modules 사용
- 토큰 기반 색상 사용
- 시안과 맞지 않는 과도한 기능 추가 금지
- 모바일 완성도는 이번 단계에서 최소 대응하되, desktop 시안 충실도를 우선
