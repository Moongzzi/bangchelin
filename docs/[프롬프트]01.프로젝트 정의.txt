# 프로젝트 정의

## 1. 프로젝트 개요
- 프로젝트명:Bangchelin Guide
- 한 줄 설명:방탈출 미식 협회 커뮤니티 웹사이트
- 프로젝트 목적:카카오톡 오픈채팅 "방탈출 미식 협회" 회원 및 방탈출 플레이어들이 이용할 수 있는 방탈출 커뮤니티용 웹사이트
- 주요 사용자:카카오톡 오픈채팅 "방탈출 미식 협회" 회원 및 방탈출 플레이어들
- 핵심 가치:방탈출 관련 커뮤니티 활성화

## 2. 프로젝트 유형
- 서비스 유형: 방탈출 유저 커뮤니티
- 플랫폼: 반응형 웹
- 렌더링 방식: Hybrid (페이지별로 다른 렌더링 방식 필요)
- 로그인 필요 여부: 일부 기능 로그인 필요
- 다국어 지원 여부: 다국어 지원 추후 고려 예정

## 3. 주요 기능
- 기능 1: 회원가입/로그인, 프로필 설정
- 기능 2: 톡방, 디스코드, 방탈출 관련 설명 Doc 문서 제공
- 기능 3: 일정 관리 시스템
- 기능 4: 문의/신고 시스템

## 4. 주요 페이지
- / : 메인 랜딩 페이지 / 히어로 섹션, CTA 버튼
- /login : 로그인 페이지 / 로그인, 회원가입 페이지 연결
- /resgister : 회원가입 페이지 / 정보 입력 필드, 회원가입 버튼
- /about : documentation 페이지 / 사이드 트리 메뉴, doc 문서
- /calendar : 일정 관리 페이지 / 메인 달력, 좌측 요약 일정 메뉴, 우측 상세 일정 패널
- /report : 문의•신고 페이지 / 문의 및 신고 작성 티켓 시스템
- /profile : 프로필 페이지 / 프로필 정보 조회 및 수정

## 5. 기술 스택
- 프레임워크: React (Vite 권장)
- 언어: TypeScript 권장
- 스타일링: Tailwind CSS 권장
- 상태관리: Zustand
- 폼 처리: React Hook Form + Zod 권장
- 데이터 패칭: Supabase REST API + fetch 또는 TanStack Query
- 라우팅: React Router
- 아이콘: Lucide React
- 애니메이션: 필요 시 Framer Motion
- 패키지 매니저: npm 또는 pnpm
- 백엔드/DB: Supabase
- API 방식: Supabase REST API

## 7. 디자인 방향
- 브랜드 성격: 프리미엄하면서도 몰입감 있는 방탈출 플랫폼. 신뢰감 있고 정돈된 정보 구조를 기반으로, 약간의 미스터리하고 고딕한 감성을 더한 브랜드 이미지
- 시각 키워드: 미슐랭 스타일, 프리미엄, 고딕, 미스터리, 몰입감, 정돈된 레이아웃, 높은 가독성, 세련된 정보 디자인
- 분위기: 전체적으로 고급스럽고 차분한 분위기를 유지하되, 방탈출 서비스에 어울리는 어두운 무드와 긴장감 있는 감성을 일부 반영. 과하게 무겁기보다는 깔끔하고 세련된 방향
- 선호 UI 스타일: 미슐랭 사이트처럼 정보가 구조적으로 잘 정리된 편집형 UI, 넓은 여백, 명확한 타이포그래피, 카드/리스트 기반 정보 배치, 직관적인 내비게이션, 콘텐츠 중심의 깔끔한 인터페이스
- 참고 서비스: 미슐랭 가이드 공식 웹사이트, 프리미엄 호텔/레스토랑 예약 서비스, 고급 매거진 스타일의 콘텐츠 플랫폼
- 피하고 싶은 스타일: 과도하게 캐주얼한 UI, 장식 요소가 많은 복잡한 화면, 과한 네온톤/게임풍 디자인, 낮은 대비로 인해 가독성이 떨어지는 스타일, 공포 연출이 지나치게 강한 다크 테마
- 접근성 고려사항: 정보량이 많은 서비스이므로 텍스트 가독성을 최우선으로 고려. 충분한 명도 대비, 명확한 타이포 위계, 읽기 쉬운 본문 크기와 행간, 모바일에서도 잘 보이는 레이아웃, 키보드 탐색 및 버튼/링크 식별성 확보

## 8. 레이아웃 규칙
- 반응형 기준: 모바일 0~767px, 태블릿 768~1023px, 데스크톱 1024~1439px, 와이드 1440px 이상
- 헤더: 상단 고정(sticky) 헤더 사용. 좌측 로고, 우측 주요 메뉴 및 CTA 배치. 바일에서는 햄버거 메뉴로 전환
- 푸터: 넉넉한 상하 여백을 가진 다단 구조 푸터 사용. 
- 사이드바: 기본 랜딩 페이지에서는 사용하지 않음. documentation 페이지에서는 좌측 고정 사이드바를 사용

## 9. 컴포넌트 전략
- 먼저 만들 공통 컴포넌트: 
    헤더, 푸터, 토글, 버튼, 인풋 필드, 드롭다운, 팝업(모달), 검색바, 라디오 버튼, 체크박스, 리스트뷰를 우선 공통 컴포넌트로 설계한다.  
    공통 컴포넌트는 전체 서비스에서 반복적으로 사용될 가능성이 높고, 스타일 일관성이 중요한 UI 요소를 중심으로 먼저 구축한다.
- 페이지 전용 컴포넌트 사용 기준: 
    특정 페이지에서만 사용되고, 다른 화면으로 재사용 가능성이 낮은 경우 페이지 전용 컴포넌트로 작성한다.  
    예를 들어 특정 상세 페이지의 전용 정보 섹션, 특정 목록 필터 패널, 특정 소개 영역 등은 해당 페이지 내부에 둔다.
- 공통화 기준: 
    2개 이상의 페이지 또는 여러 섹션에서 반복적으로 사용되며, 구조와 역할이 유사한 경우 공통 컴포넌트로 분리한다.  
    단순히 비슷해 보인다는 이유만으로 공통화하지 않고, props로 제어 가능한 범위 안에서 재사용성이 확보될 때만 분리한다.  
    UI 일관성이 중요한 입력 요소, 버튼류, 선택 요소, 오버레이 UI는 우선적으로 공통화 대상에 포함한다.
- 과도한 추상화 방지 원칙:
    초기에 너무 많은 변형을 하나의 컴포넌트에 몰아넣지 않는다.  
    재사용 패턴이 명확해진 뒤에 공통화하며, props가 지나치게 많아지거나 내부 분기가 복잡해지면 공통 컴포넌트 대신 역할별 컴포넌트로 나눈다.  
    “나중에 쓸 수도 있음”보다는 “현재 실제로 반복 사용되는가”를 기준으로 판단한다.

## 10. 데이터 전략
- 초기 mock data 사용 여부:
    사용한다.  
    초기 UI 설계와 화면 흐름 확인, 컴포넌트 단위 개발을 빠르게 진행하기 위해 mock data를 먼저 활용한다.
- mock data 위치:
    프로젝트 내 별도의 mock 또는 fixtures 디렉토리에서 관리한다.  
    페이지 전용 mock data는 해당 페이지 또는 feature 폴더 내부에 두고, 여러 화면에서 공통으로 사용하는 데이터는 공용 mock 디렉토리로 분리한다.
- API 연결 시점:
    주요 화면 구조와 공통 컴포넌트가 어느 정도 완성된 이후 Supabase REST API를 연결한다.  
    우선 mock data로 UI와 사용자 흐름을 검증한 뒤, 이후 실제 API 응답 구조에 맞춰 점진적으로 교체한다.
- loading 상태 필요 여부:
    필요하다.  
    목록, 상세, 검색 결과, 필터 적용 결과, 사용자 정보 등 비동기 데이터가 표시되는 화면에는 loading 상태를 기본적으로 고려한다.
- empty 상태 필요 여부:
    필요하다.  
    검색 결과 없음, 등록된 데이터 없음, 즐겨찾기 없음, 조건에 맞는 항목 없음 등 데이터가 비어 있는 상황을 위한 empty UI를 별도로 설계한다.
- error 상태 필요 여부:
    필요하다.  
    네트워크 오류, API 호출 실패, 권한 문제, 서버 응답 이상 등 사용자에게 안내가 필요한 상황을 고려해 에러 메시지와 재시도 UI를 제공한다.

## 11. 폴더 구조 선호
- 구조 방식:
    공통 요소는 shared 또는 components 중심으로 관리하고, 페이지별 기능은 pages 또는 features 단위로 나누는 혼합형 구조를 선호한다.  
    너무 이른 시점부터 완전한 feature-first 구조로 복잡하게 나누기보다는, 프로젝트 규모에 맞춰 점진적으로 확장 가능한 구조를 사용한다.
- 공통 컴포넌트 분리 여부:
    분리한다.  
    버튼, 입력 필드, 드롭다운, 체크박스, 라디오 버튼, 팝업, 검색바, 리스트뷰, 헤더, 푸터 등 여러 화면에서 재사용되는 컴포넌트는 공통 영역으로 분리한다.
- API 파일 분리 여부:
    분리한다.  
    API 호출 로직은 컴포넌트 내부에 직접 작성하기보다 별도 api 디렉토리 또는 feature별 api 파일로 분리해 관리한다.  
    Supabase REST API 호출 함수는 엔드포인트 단위 또는 도메인 단위로 정리한다.
- 타입 중앙 관리 여부:
    공통 타입은 중앙 관리하고, 페이지 또는 기능에 강하게 종속되는 타입은 각 feature 내부에서 관리한다.  
    전역적으로 반복 사용되는 데이터 타입(예: 사용자, 리스트 아이템, 공통 API 응답 형식)은 types 디렉토리에서 관리한다.

## 12. 코딩 규칙
- TypeScript 엄격 사용
- 가독성과 재사용성 우선
- 불필요한 의존성 추가 금지
- 하드코딩 최소화
- 토큰 기반 스타일 우선
- 접근성 기본 준수
- 반응형 고려 필수

추가 규칙:
- 하나의 컴포넌트는 하나의 역할에 집중하고, 책임이 커지면 분리할 것
- props와 상태 이름은 의미가 명확하도록 작성할 것
- 조건문과 분기 처리는 단순하게 유지하고, 복잡해지면 함수 또는 하위 컴포넌트로 분리할 것
- API 호출, 데이터 가공, UI 렌더링 로직은 가능한 한 분리할 것
- 반복되는 문자열, 설정값, 라벨, URL은 상수로 관리할 것
- 공통 컴포넌트 수정 시 기존 사용처와의 호환성을 우선 고려할 것
- 임시 코드, 사용되지 않는 코드, 불필요한 주석은 남기지 않을 것
- 스타일은 기존 디자인 토큰과 레이아웃 규칙 안에서 작성할 것

## 13. AI 출력 규칙
- 코드 생성 전 구현 계획을 간단히 설명할 것
- 파일 경로를 명확히 표시할 것
- 한 번에 전체 구현하지 말고 단계적으로 진행할 것
- 기존 컴포넌트를 우선 재사용할 것
- 요구사항이 애매하면 질문할 것
- 관련 없는 파일은 수정하지 말 것

추가 규칙:
- 새 컴포넌트가 필요한 경우 기존 컴포넌트로 대체 가능한지 먼저 검토할 것
- 스타일 수정 시 기존 토큰과 레이아웃 규칙을 우선 따를 것
- mock data 단계와 실제 API 연결 단계를 혼합하지 말 것
- any 사용을 피하고 명확한 타입을 사용할 것
- 기존 코드 수정 시 변경 이유와 영향 범위를 함께 설명할 것
- 예시 코드는 실행 가능한 형태로 작성하고, 생략 시 그 범위를 명확히 표시할 것

## 14. 우선 구현 순서
1. 디자인 토큰과 기본 스타일 가이드 정리  
   색상, 타이포그래피, 간격, radius, shadow 등 기본 토큰을 먼저 정의하고 전체 UI의 기준을 마련한다.
2. 공통 컴포넌트 우선 구현  
   헤더, 푸터, 버튼, 인풋 필드, 드롭다운, 토글, 라디오 버튼, 체크박스, 팝업, 검색바, 리스트뷰 등 자주 사용하는 공통 컴포넌트를 먼저 만든다.
3. mock data 기반으로 주요 페이지 UI 구성  
   실제 API 연결 전 mock data를 사용해 메인 페이지, 목록 페이지, 상세 페이지 등 핵심 화면의 레이아웃과 사용자 흐름을 먼저 완성한다.
4. 상태 처리 및 예외 UI 정리  
   loading, empty, error 상태를 포함해 검색, 필터, 선택, 팝업 등 상호작용 흐름을 정리하고 사용자 경험을 다듬는다.
5. Supabase REST API 연결 및 데이터 연동  
   mock data로 검증한 화면과 컴포넌트에 실제 API를 연결하고, 응답 구조에 맞춰 데이터 바인딩과 예외 처리를 마무리한다.
