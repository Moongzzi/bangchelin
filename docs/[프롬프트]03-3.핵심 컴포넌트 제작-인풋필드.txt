[이번 캡처 기준으로 확인되는 패턴]
반복되는 요소라고 생각하는 것:
- 상단 label
- input control
- placeholder/value 텍스트
- underline 또는 box border
- 우측 clear icon button
- 하단 helper/error/success message
- 상태별 컬러 변화
- 공통 세로 스택 구조

이번 구현에서 우선 반영할 8가지 variation/state:
1. default
   - label 있음
   - underline 스타일
   - placeholder 표시
2. default-with-value
   - label 있음
   - 값 존재
   - clear 버튼 표시
3. focused
   - label 있음
   - 포커스 상태
   - 커서 표시 또는 active underline 강조
4. disabled
   - label 있음
   - 선택 또는 입력 불가 상태
   - 일반 입력 상태와 구분되는 비활성 스타일
   - 배경 또는 텍스트/보더 톤 변화 가능
5. error
   - label 있음
   - 값 존재 가능
   - clear 버튼 표시 가능
   - 하단 에러 메시지 표시
6. success
   - label 있음
   - 값 존재 가능
   - clear 버튼 표시 가능
   - 하단 성공 메시지 표시
7. outlined
   - label 있음
   - 사각형 border 스타일
   - 내부에 placeholder 표시
8. bare / minimal
   - 최소형 표현
   - label 없이 input만 보이거나
   - 혹은 하단 underline만 있는 단순 표현

애매하거나 아직 미정인 부분:
- exact height / padding / spacing 값
- clear 버튼 hover/focus 스타일
- 각 상태별 border 두께 변화
- 커서 색상
- disabled 상태에서 배경색/텍스트색 처리 방식의 정확한 수치
- readOnly 상태를 disabled와 분리할지 여부

[레이아웃/구조 해석]
이미지에서 명확히 확인 가능한 사실:
- 대부분의 인풋은 상단 라벨 + 하단 입력영역 구조임
- underline 기반 표현이 여러 번 반복됨
- 일부 상태에는 오른쪽 원형 clear 버튼이 있음
- 일부 상태에는 하단에 컬러 메시지가 있음
- 하나의 상태는 일반 입력형보다 비활성처럼 보이는 필드 표현임
- 하나의 상태는 외곽선이 있는 outlined 박스 형태임
- 전체적으로 세로로 나열된 상태 모음 시안임

텍스트 설명으로 보강된 사실:
- 에러 상태 메시지는 검증 실패를 나타냄
- 성공 상태 메시지는 검증 성공 또는 사용 가능 상태를 나타냄
- clear 버튼은 입력값 초기화 역할임
- bare/minimal 형태는 간단한 입력 문맥에서 사용 가능함

구현을 위해 합리적으로 추정하는 사실:
- InputField 루트 아래 label, control, message 구조가 적절함
- clear 버튼은 값이 있을 때만 조건부 렌더링하는 구조가 적절함
- variant와 validation state를 분리하면 재사용성이 높음
- visual variant(underline/outlined)와 semantic state(default/focused/disabled/error/success)를 함께 다룰 수 있어야 함
- 다만 이번 단계에서는 시안 기준 8개 표현을 우선 구현하고, 과도한 상태 조합은 만들지 않는 것이 적절함

[레이아웃/상태 규칙]
인풋 타입:
우선 text 기준으로 구현하되, input 태그의 기본 type 확장은 가능하게 설계

정렬 규칙:
- label은 control 위에 위치
- 메시지는 control 아래에 위치
- clear 버튼은 control 우측 내부 정렬
- 메시지는 좌측 정렬

상태 처리 방향:
- visual variant와 validation/message 상태를 props로 제어 가능하게 설계
- 단, 사용 예시는 시안 기준 8가지 표현을 중심으로 제공
- focus는 :focus-visible 및 state class 모두 대응 가능하게 고려
- disabled는 HTML disabled 속성과 시각 스타일이 함께 반영되도록 설계
