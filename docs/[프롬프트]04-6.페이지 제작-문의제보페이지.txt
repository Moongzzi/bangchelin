당신은 React + TypeScript 기반 프로젝트의 시니어 프론트엔드 엔지니어이자,
기존에 생성된 공통 컴포넌트와 페이지 구조를 정리하면서
실제 서비스 페이지를 완성하는 UI 구현 어시스턴트입니다.

현재 프로젝트는 공통 컴포넌트(예: Header, Footer, InputField, Dropdown, Button, Popup/Modal 등)가
어느 정도 제작된 상태이며,
기존 Agent가 일부 페이지를 임의로 생성했을 가능성도 있습니다.

이번 작업의 목적은 문의/제보 페이지를 구현하는 것입니다.

이 페이지는 사용자가 문의 또는 제보 내용을 작성해서 제출할 수 있는 페이지이며,
중요한 요구사항은 다음과 같습니다.

[핵심 요구사항]
1. 사용자가 문의/제보 페이지에 진입하면,
   먼저 “임시 저장된 내용이 있는지” 조회해야 합니다.
2. 이 조회는 추후 API로 연동될 예정이므로,
   현재는 mock 함수 / placeholder API / mock hook으로 구조만 먼저 설계하세요.
3. 임시 저장된 내용이 있으면,
   “저장된 내용을 불러올지” 묻는 팝업을 먼저 노출해야 합니다.
4. 사용자가 불러오기를 선택하면,
   저장된 내용을 폼에 채워 넣습니다.
5. 사용자가 불러오지 않겠다고 하면,
   빈 상태로 작성할 수 있게 합니다.
6. 저장된 내용이 아예 없다면,
   팝업 없이 바로 빈 폼 상태로 작성할 수 있게 합니다.
7. 페이지 내부에는 최소한 다음 폼 요소가 있어야 합니다.
   - 닉네임
   - 문의 종류
   - 문의 제목
   - 문의 내용
8. 페이지 하단에는 “임시 저장” 버튼과 “보내기” 버튼이 있어야 합니다.
9. 임시 저장 성공 시 완료 팝업을 띄울 수 있도록 설계하세요.
10. 보내기 완료 시 전송 완료 팝업을 띄울 수 있도록 설계하세요.
11. 전체 구조는 실제 API가 붙지 않아도 동작 흐름이 보이도록 만들어야 합니다.
12. 이미지를 참고하되, 실제 수치 구현은 내가 아래에 적을 설계값과 디자인 토큰을 우선 적용해야 합니다.

중요:
이번 작업은 이미지를 보고 감으로 수치를 정하는 작업이 아닙니다.
내가 아래에 입력하는 실제 사이즈, spacing, color token, radius, typography, 버튼 규격 등을 최우선 적용하세요.
값이 비어 있는 경우에만 합리적으로 추정하고, 반드시 추정값이라고 명시하세요.

또한 기존 Agent가 InquiryPage.tsx, ReportPage.tsx, ContactPage.tsx 또는 유사 파일을 이미 만들어두었을 가능성이 있으므로,
무조건 새로 만드는 방식보다 점검/수정/리팩터링 관점으로 접근하세요.

[시안 해석]
이미지에서 확인 가능한 사실:
- 상단에는 공통 헤더가 있음
- 페이지 중앙 상단에 “문의하기” 제목이 있음
- 그 아래 첫 줄에는 닉네임 입력 필드와 문의 종류 선택 필드가 있음
- 메인 폼 박스 안에는 문의 종류, 문의 제목, 문의 내용 입력 영역이 있음
- 문의 내용은 textarea 형태로 보임
- 하단 오른쪽에는 “임시 저장” 버튼과 “보내기” 버튼이 있음
- 팝업 시안이 별도로 존재하며,
  - 저장된 내용을 불러올지 묻는 팝업
  - 임시 저장 완료 팝업
  - 전송 완료 팝업
  이 있는 것으로 보임
- 하단에는 공통 푸터가 있음

텍스트 요구사항으로 확정된 내용:
- 화면 진입 시 임시 저장된 내용이 있는지 조회
- 이 조회는 추후 API 연동 예정
- 임시 저장된 내용이 있다면 불러올 건지 묻는 팝업 노출
- 저장된 내용이 없다면 바로 작성 가능
- 임시 저장 및 전송 완료 팝업도 고려
- 페이지는 문의/제보 작성 흐름을 제공해야 함

구현을 위해 합리적으로 추정 가능한 사항:
- 문의 종류는 드롭다운으로 처리하는 것이 적절함
- 문의/제보는 하나의 페이지에서 타입만 달리 처리할 가능성이 있음
- draft 조회는 useEffect + mock async 함수 구조로 잡아두면 추후 API 교체가 쉬움
- draft 팝업은 진입 직후 조건부 전시되는 modal/popup 형태가 적절함
- 보내기/임시 저장은 실제 API 대신 mock submit 함수와 상태값으로 대체 가능
- 완료 팝업은 공통 Modal/Popup 컴포넌트 재사용이 적절함

[중요 구현 원칙]
1. 첨부 이미지는 참고용이며, 실제 수치는 내가 입력하는 설계값을 우선 적용하세요.
2. 기존 Agent가 유사 페이지를 만들어뒀다면 리팩터링 관점으로 접근하세요.
3. 페이지 진입 시 draft 존재 여부 확인 흐름을 가장 먼저 설계하세요.
4. draft가 있으면 “복원할지 묻는 팝업”이 먼저 보여야 합니다.
5. draft가 없으면 바로 빈 폼 상태로 진입해야 합니다.
6. draft 복원 여부를 선택하기 전까지는 사용자가 폼을 바로 수정하지 않는 구조를 제안해도 됩니다.
7. 임시 저장 / 전송 / 완료 팝업까지 실제 동작 흐름이 보이는 상태 구조를 설계하세요.
8. 아직 API가 없으므로 mock async 함수 / hook / service 계층을 분리하세요.
9. 디자인 토큰은 가능한 CSS 변수 또는 tokens/constants로 정리하세요.
10. 공통 InputField, Dropdown, Button, Modal이 있다면 최대한 재사용하세요.
